Syllabus2Calendar

Web application that automates college course syllabi organization

1 / 1

Problem

At the beginning of each semester, students manually create hundreds of organization objectives to stay on top of coursework throughout the semester. This process is way too time consuming, but necessary to succeed as a student. The manual process also lacks optimization features modern college studnts need..

Result

An in-browser flow from pasted or uploaded syllabi to downloadable ICS files with configurable reminders and report-style summaries.

Core components

  • User inputs: Syllabus text upload, class schedule pattern (e.g. MWF), API key, reminder windows, report preferences.
  • System processes: LLM-backed parse and normalization; multi-format date handling; event classification (homework, exam, project, etc.); ICS generation.
  • Outputs: One ICS file with thousands of organized event objects per class, ready for Google Calendar, Apple Calendar, and Outlook, as well as any alternative calendar applications that support ICS. Email reports with upcoming objectives and dates.

What I learned

I had to be clever in how I built this under a tight deadline. Originally I'd built this seeing it as a challenge and also out of senoritis, it was either I automated my course planning or I was going to be missing some homework assignments this semester. At first just the backend, the functions to make it run. Then, in one of my courses the project had applicability and I decided to try getting creative in the frontend, hence the Windows XP style. It's a minimum viable product using a supply chain of services including APIs and utilizes common web services to make it easy to deploy with the ICS file output. I think a next step could have been building a database system to fulfill an ecosystem around the proejct, calendar conventions are limited and I think there's opportunity there. Overall it was an ad hoc project built on real user case with innovative intersecting of various technologies for a fast and efficient pipeline solution to a widespread problem.

Problem

Outbound and sales teams need trustworthy, detailed restaurant profiles—hours, services, photos, proximity to existing clients—before a call. Pulling that together by hand across maps sites, review sources, and spreadsheets is slow and inconsistent at volume.

Result

Full‑stack app that enriches restaurant leads into sales‑ready profiles, streaming progress to a dashboard.

Core components

  • User inputs: CSV upload for client locations and batch lead enrichment; configured Google Places, Custom Search Engine, Vision, and optional Maps keys.

Reader Suite

Desktop application for capturing and generating digital books, as well as a library suite designed for mobile, offline reading and synchronized access across devices.

1 / 1

Problem

Readers need an accessible, low-cost solution to engage with digital content. Power is limited, and much media is locked behind paywalls or other restrictions. Many users want a digital library, but current solutions do not make it easy.

Result

An end-to-end pipeline from desktop capture to ebooks, a progressive web application library for offline reading, and an optional backend for cross-device synchronization.

Core components

  • User inputs: Source material and capture tool usage.
  • System processes: Automated ebook generation. Encrypted formatting and storage. Reader parsing and rendering.
  • Outputs: Portable ebooks; offline-first reading; cross-device library.

What I learned

This started with a simple idea—to read anything I want, wherever and whenever I want. Through building it, I learned a lot across the stack: custom file formats, encryption and key handling, progressive web apps, interaction automation, to name a few. I even designed two different navigation systems (scroll and click) and worked through layering and event handling to simulate input reliably. This was the first time I've actually worked on a project that I'd use so often formyself, and I learned about the end-to-end design process. I also experimented with setting up a PostgreSQL server for syncing across devices. Getting it to run on my iPad and being able to lay back and read a book I captured was pretty cool.
